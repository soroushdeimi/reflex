## توضیحات
پروتکل Reflex یک پروتکل پراکسی مدرن برای Xray-Core است که با تمرکز بر امنیت بالا و مخفی‌کاری (Stealth) طراحی شده است. این پیاده‌سازی شامل تمامی ۵ مرحله درخواستی و قابلیت‌های پیشرفته است.

### ویژگی‌های کلیدی:
- **هندشیک ضمنی (Implicit Handshake):** پشتیبانی از تشخیص هوشمند از طریق Magic Number یا درخواست‌های شبیه HTTP POST.
- **امنیت پیشرفته:** تبادل کلید X25519، تولید کلیدهای نشست مجزا برای جلوگیری از Nonce Reuse و رمزنگاری ChaCha20-Poly1305.
- **محافظت در برابر Replay:** استفاده از فیلتر BloomRing و اعتبارسنجی Timestamp برای رد کردن درخواست‌های تکراری.
- **سیستم Fallback هوشمند:** هدایت ترافیک غیرمجاز به وب‌سرور مقصد بدون از دست دادن بایت‌های اولیه با استفاده از تکنیک Peeking.
- **شکل‌دهی ترافیک (Traffic Morphing):** شبیه‌سازی آماری الگوهای ترافیکی (مانند یوتیوب) با استفاده از توزیع‌های وزنی برای سایز بسته‌ها و تاخیرها.
- **فریم‌های کنترلی:** پشتیبانی از PADDING_CTRL و TIMING_CTRL برای بروزرسانی پویا مشخصات ترافیک در طول نشست.

## نحوه اجرا و پیکربندی
### پیش‌نیازها
- یک سرور لینوکس (مثلاً Ubuntu 22.04) با IP ثابت.
- یک نام دامنه (Domain) که به IP سرور اشاره می‌کند. (در صورت فعالسازی tls)
- پورت ۴۴۳ و ۸۰ باز روی سرور.

### ۱. بیلد کردن پروژه
ابتدا باید فایل اجرایی Xray را با کدهای جدید بسازید:
```bash
cd xray-core
go build -o xray ./main
```

### ۲. تنظیمات سرور (Inbound)
یک فایل به نام `server.json` بسازید. در اینجا نمونه‌ای با تمام قابلیت‌ها آمده است:

```json
{
  "log": {
    "loglevel": "info"
  },
  "inbounds": [
    {
      "port": 443,
      "protocol": "reflex",
      "settings": {
        "clients": [
          {
            "id": "ad600f1c-7798-43e8-8319-33b1e3e7843d",
            "policy": "youtube"
          }
        ],
        "fallback": {
          "dest": 80
        },
        "tls": {
        "enabled": true,
        "server_name": "yourdomain.com",
        "cert_file": "/etc/letsencrypt/live/yourdomain.com/fullchain.pem",
        "key_file": "/etc/letsencrypt/live/yourdomain.com/privkey.pem",
        "ech_key": "BASE64_ENCODED_ECH_PRIVATE_KEY_SET"
      }
      }
    }
  ],
  "outbounds": [
    {
      "protocol": "freedom",
      "tag": "direct"
    }
  ]
}

```
سپس دستور زیر:
```bash
./xray run -c server.json
```

### ۳. تنظیمات کلاینت (Outbound)
یک فایل به نام `client.json` بسازید:

```json
{
  "log": {
    "loglevel": "info"
  },
  "inbounds": [
    {
    "port": 1080,
    "protocol": "socks",
    "settings": {
      "auth": "noauth"
    }
  }
  ],
  "outbounds": [
    {
      "protocol": "reflex",
      "settings": {
        "address": "Server_IP",
        "port": 443,
        "id": "ad600f1c-7798-43e8-8319-33b1e3e7843d",
        "tls": {
          "enabled": true,
          "server_name": "yourdomain.com",
          "ech_key": "BASE64_ENCODED_ECH_PUBLIC_KEY"
        }
      },
      "tag": "proxy"
    },
    {
      "protocol": "freedom",
      "tag": "direct"
    }
  ]
}
```
سپس دستور زیر:
```bash
./xray run -c client.json
```


### ۳. تست
در سرور کلاینت و در یک ترمینال مجزا با دستور زیر میتوانید از سرویس تست بگیرید، در صورت سلامت سیستم باید آیپی سرور را چاپ کند:
```bash
curl -x socks5://localhost:1080 api64.ipify.org
```

## راهنمای تنظیمات پیشرفته

### مکانیزم Fallback (ترافیک فریب)

پروتکل Reflex دارای سیستم تشخیص هوشمند است. هنگامی که یک کانکشن به پورت ورودی (مثلاً 443) متصل می‌شود، هندلر Reflex ابتدا ۶۴ بایت اول را به صورت Peek بررسی می‌کند:

1.  **بررسی Magic Number:** اگر ۴ بایت اول برابر با `0x52454658` باشد، آن را به عنوان پروتکل Reflex شناسایی می‌کند.
2.  **بررسی HTTP POST:** اگر با کلمات "POST" شروع شود (برای تونل‌های مبتنی بر HTTP)، آن را به عنوان Reflex شناسایی می‌کند.
3.  **هدایت به Fallback:** اگر هیچ‌کدام از موارد بالا مطابقت نداشته باشد یا احراز هویت کاربر با شکست مواجه شود، کانکشن بدون قطع شدن به آدرس مقصد تعریف شده در `fallback` هدایت می‌شود.

- اجرای وب سرور ساده nginx روی پورت ۸۰:
```bash
sudo apt install -y nginx
sudo systemctl start nginx
```

### سیاست‌های شکل‌دهی ترافیک (Policy & Morphing)
مورفینگ در پروتکل Reflex از توزیع‌های وزنی (Weighted Distribution) استفاده می‌کند. برای هر پروفایل، لیستی از سایزها و تاخیرهای محتمل با وزن‌های مشخص تعریف شده است. در هنگام ارسال داده، یک مقدار به صورت تصادفی بر اساس این وزن‌ها انتخاب می‌شود تا الگوهای ثابت ترافیک شکسته شود.

با تنظیم `policy` در بخش کلاینت‌های سرور، می‌توانید الگوی ترافیک و سرعت را تغییر دهید:
- **streaming:** بالاترین سرعت ممکن (مناسب برای دانلود و استریم 4K). از پکت‌های بزرگ (تا ۶۴ کیلوبایت) استفاده می‌کند.
- **youtube:** تعادل بین مخفی‌کاری و سرعت (شبیه‌سازی الگوی یوتیوب).
- **web:** برای وب‌گردی عادی با پکت‌های کوچک‌تر.
- **بدون Policy:** غیرفعال کردن مورفینگ برای کمترین تاخیر ممکن (مناسب گیمینگ).

### تنظیم TLS و ECH در دنیای واقعی

برای امنیت حداکثری و پنهان کردن SNI، پروتکل Reflex از TLS 1.3 و ECH پشتیبانی می‌کند.

#### ۱. دریافت گواهی معتبر با Certbot (مانند Let's Encrypt)
در دنیای واقعی، استفاده از Self-signed certificate باعث شناسایی سریع می‌شود. بهتر است از Certbot استفاده کنید:
```bash
sudo apt install certbot
sudo certbot certonly --standalone -d yourdomain.com
```

#### ۲. تولید کلیدهای ECH
برای فعال‌سازی ECH، سرور به یک جفت کلید نیاز دارد.
- **کلید عمومی (ECH Config):** برای کلاینت جهت رمزنگاری SNI.
- **کلید خصوصی (KeySet):** برای سرور جهت رمزگشایی.
- جهت ساخت کلید ها از اسکریپت زیر استفاده کنید.
```bash
cd xray-core
go run gen_ech.go yourdomain.com
```

#### ۳. فعال‌سازی در کانفیگ
- **cert_file & key_file:** آدرس فایل‌های تولید شده توسط Certbot را بدهید.
- **ech_key:** طبق خروجی مشخص شده در اسکریپت gen_ech.go در کلاینت و سرور جایگذاری شود.
- **Morphing:** با تنظیم `policy` به `youtube` ترافیک TLS شما حتی از نظر رفتار پکت‌ها شبیه به ترافیک یوتیوب می‌شود که شناسایی آن را تقریباً غیرممکن می‌کند.

## تست‌ها
- **واحد (Unit Tests):** پوشش کامل توابع رمزنگاری، مدیریت فریم و توزیع‌های آماری.
- **یکپارچگی (Integration Tests):** تست سناریوهای اتصال کامل، فال‌بک، حملات Replay و هندشیک HTTP.
- **پوشش کد (Coverage):** بیش از ۶۵٪ پوشش کد در پکیج‌های مربوطه.
- **Race Detection:** تمامی تست‌ها با ابزار تشخیص تداخل حافظه پاس شده‌اند.

## عیب‌یابی (Debug)
اگر اتصال برقرار نمی‌شود:
1. **زمان سیستم:** اختلاف ساعت سرور و کلاینت نباید بیش از ۵ دقیقه باشد (به دلیل مکانیزم جلوگیری از Replay).
2. **گواهی TLS:** مطمئن شوید فایل‌های گواهی توسط Xray قابل خواندن هستند.
3. **لاگ‌ها:** برای مشاهده جزئیات خطا، لاگ‌های Xray را بررسی کنید:
   ```bash
   tail -f /var/log/xray/access.log
   tail -f /var/log/xray/error.log
   ```
4. **فایروال:** پورت ۴۴۳ (TCP) باید روی سرور باز باشد.

## مشکلات و راه‌حل‌ها
# مشکلات و راه‌حل‌های پروژه Reflex

در طول پیاده‌سازی پروتکل Reflex، با چالش‌های فنی متعددی روبرو شدیم که هر کدام با راهکارهایی حل شدند. در ادامه لیست برخی از این موارد آورده شده است:

۱. **مشکل Nonce Reuse در رمزنگاری:**
   - **مشکل:** استفاده از یک کلید برای هر دو جهت (رفت و برگشت) در ChaCha20-Poly1305 منجر به استفاده مجدد از نانس (Nonce Reuse) و به خطر افتادن کامل امنیت استریم می‌شد.
   - **راه‌حل:** با استفاده از تابع HKDF، دو کلید مجزای `c2s` (کلاینت به سرور) و `s2c` (سرور به کلاینت) از کلید مشترک (Shared Secret) استخراج شد تا امنیت در هر دو جهت تضمین شود.

۲. **از دست رفتن بایت‌های اول در Fallback:**
   - **مشکل:** هنگام استفاده از `Peek` برای تشخیص پروتکل (تشخیص Magic Number یا HTTP POST)، بایت‌هایی که خوانده شده بودند در حافظه میانی (Buffer) باقی می‌ماندند و در صورت هدایت به سرور Fallback، این بایت‌ها به مقصد ارسال نمی‌شدند که باعث شکست ارتباط HTTP می‌شد.
   - **راه‌حل:** یک ساختار `PreloadedConn` پیاده‌سازی شد که بایت‌های Peek شده را در ابتدای استریم باز می‌گرداند و سپس داده‌های زنده را از کانکشن اصلی می‌خواند.

۳. **خطای Malformed ECHConfigList:**
   - **مشکل:** برخی ابزارهای تولید کلید ECH، پیشوند طول (Length Prefix) که شامل ۲ بایت است را به ابتدای Config اضافه نمی‌کنند. این موضوع باعث می‌شد کتابخانه TLS در سمت کلاینت با خطای Malformed مواجه شود.
   - **راه‌حل:** در هندلر Outbound، مکانیزمی طراحی شد که به صورت هوشمند نسخه ECH (مانند 0xfe0d) را تشخیص داده و در صورت نبود پیشوند طول، آن را به صورت خودکار محاسبه و به ابتدای تنظیمات اضافه می‌کند.

۴. **کاهش سرعت در ترافیک سنگین (Morphing):**
   - **مشکل:** استفاده از الگوریتم‌های شکل‌دهی ترافیک با تاخیرهای ثابت (Fixed Delays) باعث افت شدید سرعت دانلود و آپلود (Throttling) می‌شد.
   - **راه‌حل:** پروفایل اختصاصی `streaming` طراحی شد که از فریم‌های بزرگ ۶۴ کیلوبایتی و تاخیرهای بسیار کوتاه در مقیاس میکروثانیه استفاده می‌کند تا در عین حفظ الگوی آماری، حداکثر پهنای باند را ارائه دهد.

۵. **عدم تولید نانس تصادفی در هندشیک:**
   - **مشکل:** در نسخه‌های اولیه، نانس مورد استفاده در هندشیک کلاینت مقداری ثابت یا پیش‌بینی‌پذیر داشت که ریسک حملات Replay را افزایش می‌داد.
   - **راه‌حل:** کلاینت موظف شد برای هر اتصال جدید، یک Nonce کاملاً تصادفی ۱۲ بایتی تولید و ارسال کند. همچنین در سمت سرور از Bloom Filter برای جلوگیری از پردازش نانس‌های تکراری استفاده شد.

۶. **ناهماهنگی در تگ‌های JSON تنظیمات:**
   - **مشکل:** استفاده از CamelCase در تگ‌های Structهای Go باعث می‌شد فایل‌های تنظیمات استاندارد Xray (که از snake_case استفاده می‌کنند) به درستی پارس نشوند.
   - **راه‌حل:** تمام تگ‌های JSON در بخش `infra/conf` به `snake_case` تغییر یافت تا با اکوسیستم Xray سازگاری کامل داشته باشد.

۷. **خطای TLS Unrecognized Name:**
   - **مشکل:** در اتصالاتی که SNI مشخصی نداشتند، هندشیک TLS به دلیل عدم تطابق نام سرور با شکست مواجه می‌شد.
   - **راه‌حل:** منطق انتخاب `ServerName` در Outbound بهبود یافت تا در صورت نبود SNI در تنظیمات، از آدرس مقصد (Destination Address) یا مقادیر پیش‌فرض هوشمند استفاده کند.

۸. **فرمت ناسازگار کلید خصوصی ECH:**
   - **مشکل:** کتابخانه `xtls` برای کارکرد صحیح ECH به فرمت خاصی از کلید (ترکیب کلید خصوصی و کانفیگ با پیشوند طول) نیاز داشت که با فرمت‌های PEM معمولی متفاوت بود.
   - **راه‌حل:** یک اسکریپت Go بنام `gen_ech.go` ارائه شد که کلیدهای خام را دریافت کرده و فرمت باینری دقیق مورد نیاز Xray را تولید می‌کند.

۹. **مشکل همگام‌سازی استریم در فریم‌های حجیم:**
   - **مشکل:** ارسال داده‌های حجیم در یک مرحله گاهی باعث می‌شد که مرز فریم‌ها در سمت گیرنده به درستی تشخیص داده نشود.
   - **راه‌حل:** منطق `Write` بازنویسی شد تا داده‌ها را به قطعات حداکثر ۶۴ کیلوبایتی تقسیم کرده و هر قطعه را با هدر و تگ احراز هویت (MAC) مجزا ارسال کند.

۱۰. **عدم باز شدن وب‌سایت‌ها روی برخی شبکه‌های موبایل (MTU/MSS):**
    - **مشکل:** روی برخی گوشی‌ها و اپراتورهای موبایل، وب‌سایت‌های سنگین (مثل Google) باز نمی‌شدند در حالی که اتصال پراکسی برقرار بود.
    - **راه‌حل:** مشخص شد که افزودن هدرهای پروتکل Reflex و TLS باعث بزرگتر شدن سایز پکت‌ها از حد مجاز شبکه (MTU) می‌شود. با کاهش سایز پیش‌فرض پکت‌ها در پروفایل‌های مورفینگ به ۱۳۵۰ بایت و ارائه راهنمای تنظیم MSS در سمت کلاینت، این مشکل برطرف گردید.

۱۱. **خطای Can't Resolve Host (عدم پشتیبانی از UDP):**
    - **مشکل:** کاربران در هنگام استفاده از پروتکل با خطای عدم حل نام دامنه مواجه می‌شدند زیرا پروتکل فقط از TCP پشتیبانی می‌کرد و کوئری‌های DNS (که مبتنی بر UDP هستند) با شکست مواجه می‌شدند.
    - **راه‌حل:** با افزودن فیلد Network به فریم اولیه و پیاده‌سازی مکانیزم کپسوله‌سازی UDP-over-TCP، پشتیبانی کامل از ترافیک UDP و DNS به پروتکل Reflex اضافه شد.
