# پروژه Reflex - متین زارع بیدکی و حسین انجیدنی

## شماره دانشجویی

- متین زارع بیدکی 401100166
- حسین انجیدنی: 400100746

## توضیحات

این پروژه پیاده‌سازی پروتکل Reflex در Xray-Core است. پیاده‌سازی با دنبال کردن گام‌به‌گام مستندات موجود در پوشه `docs` انجام شده است. ابتدا ساختار اولیه پروتکل و تعریف config.proto پیاده‌سازی شد. سپس handshake مبتنی بر magic number با استفاده از key exchange و authentication کاربران اضافه شد. در مرحله بعد، رمزنگاری با ChaCha20-Poly1305 و ساختار Frame برای انتقال داده‌های رمزنگاری شده پیاده‌سازی شد. قابلیت fallback برای هدایت ترافیک غیر-Reflex به سرور fallback اضافه شد که با استفاده از `Peek` برای تشخیص نوع ترافیک کار می‌کند.

یکی از قابلیت‌های پیشرفته که پیاده‌سازی شد، Traffic Morphing است. این قابلیت ترافیک را به گونه‌ای شکل می‌دهد که شبیه پروتکل‌های بی‌خطر مثل HTTP/2 API به نظر برسد. Morphing شامل کنترل توزیع آماری اندازه بسته‌ها و timing بین بسته‌ها است. این قابلیت ابتدا در inbound پیاده‌سازی شد، اما بعد از پیاده‌سازی outbound، مشکلاتی در هماهنگی بین دو طرف ایجاد شد که نیاز به اصلاح داشت.

پروتکل Reflex شامل دو بخش اصلی است: inbound (سمت سرور) که الزامی است و outbound (سمت کلاینت) که اختیاری است. در این پیاده‌سازی هر دو بخش تکمیل شده‌اند. در طول پیاده‌سازی تغییرات و دیباگ‌های لازم برای سازگاری با ساختار Xray-Core اعمال شد.

## نحوه اجرا

برای اجرای پروژه، ابتدا باید Xray-Core را با فایل کانفیگ سرور اجرا کنید. فایل `xray-core/server_config.json` شامل تنظیمات inbound برای پروتکل Reflex است که روی پورت 8081 گوش می‌دهد.

برای اتصال به سرور، باید یک نمونه دیگر از Xray-Core را با فایل `xray-core/client_config.json` اجرا کنید. این فایل یک SOCKS5 proxy روی پورت 1080 و آدرس 127.0.0.1 باز می‌کند که می‌توانید از طریق افزونه‌های مرورگر مثل FoxyProxy به آن متصل شوید.

برای تست قابلیت fallback، می‌توانید از فایل `xray-core/config-fallback-test.json` استفاده کنید که شامل تنظیمات fallback به پورت 8080 است. نحوه کار fallback به این صورت است که وقتی یک اتصال به سرور Reflex برقرار می‌شود، سرور ابتدا با استفاده از `Peek` روی 64 بایت اول، نوع ترافیک را تشخیص می‌دهد. اگر ترافیک شامل magic number یا handshake Reflex نباشد، سرور تمام اتصال را به سرور fallback (که در کانفیگ مشخص شده) فوروارد می‌کند. برای تست این قابلیت، می‌توانید سرور fallback ساده‌ای مثل `docs/fallback_server.py` را اجرا کنید که یک سرور HTTP ساده روی پورت 8080 است. این سرور فقط برای تست است و در پیاده‌سازی واقعی نیازی به آن نیست - هر سرور HTTP یا وب‌سایت معمولی می‌تواند به عنوان fallback استفاده شود.

## مشکلات و راه‌حل‌ها

یکی از مشکلات اصلی این بود که در ابتدا outbound را پیاده‌سازی نکرده بودیم و فقط روی inbound تمرکز داشتیم. بعد از تکمیل inbound، تصمیم گرفتیم outbound را هم پیاده‌سازی کنیم تا پروژه کامل‌تر شود.

در پیاده‌سازی outbound، با مشکل مهمی در مورد morphing مواجه شدیم. نمی‌توانستیم تشخیص دهیم که آیا بسته‌ها در سمت outbound morph شده‌اند یا نه. این مشکل باعث خرابی داده‌ها در ارتباط end-to-end می‌شد. برای حل این مشکل، از یک flag به نام `morphing_enabled` استفاده کردیم و بررسی‌های سخت‌گیرانه‌ای اضافه کردیم تا مطمئن شویم که morphing فقط زمانی اعمال می‌شود که در هر دو طرف (inbound و outbound) فعال باشد. این راه‌حل مشکل corruption داده‌ها را برطرف کرد و ارتباط end-to-end به درستی کار می‌کند.

در طول پیاده‌سازی handshake بین کلاینت و سرور، با مشکلاتی در parsing پاسخ HTTP مواجه شدیم. سرور پاسخ handshake را به صورت HTTP 200 OK با یک JSON body که شامل public key است ارسال می‌کند. در سمت کلاینت، باید این پاسخ را به درستی parse کنیم تا public key را استخراج کنیم. مشکل این بود که اگر پاسخ HTTP به درستی فرمت نشده باشد یا headerها کامل نباشند، parsing با خطا مواجه می‌شد. برای حل این مشکل، بررسی‌های دقیق‌تری برای خواندن status line و headerها اضافه کردیم و از `bufio.Reader` برای خواندن خط به خط استفاده کردیم تا مطمئن شویم که تمام headerها به درستی خوانده می‌شوند.

همچنین در ابتدای کار با مشکل `Peek` در تشخیص پروتکل مواجه شدیم. سرور باید حداقل 64 بایت از ابتدای اتصال را peek کند تا بتواند نوع ترافیک را تشخیص دهد. اگر ترافیک کمتر از 64 بایت باشد، `Peek` منتظر می‌ماند و ممکن است باعث timeout شود. این مشکل به خصوص در تست fallback خود را نشان داد. برای حل این مشکل، مطمئن شدیم که در تست‌ها حداقل 64 بایت داده ارسال می‌شود و در پیاده‌سازی واقعی، اگر داده کافی نباشد، سرور به fallback می‌رود.
